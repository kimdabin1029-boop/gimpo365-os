## 1. 문서 목적과 성격

이 문서는 김포365OS 작업 후 **사람이 직접 눈으로 확인하는 수동 점검표**다.

성격을 분명히 한다.

```text
이 문서는 OS 공통 smoke test 및 회귀(regression) 점검표다.
상세 기능 QA 문서가 아니다.
자동 테스트(python manage.py check / test)를 대체하지 않는다. 그 다음에 하는 사람의 눈 확인이다.
Inventory Module의 상세 QA는 docs/modules/inventory/INVENTORY_MANUAL_QA_CHECKLIST.md가 담당한다.
```

목적: OS 틀 작업·새 모듈 추가·migration이 **기존에 잘 되던 것을 깨뜨리지 않았는지**를 빠르게 확인한다.

---

## 2. 점검 전 준비

```text
[ ] .venv 활성화됨 ( 프롬프트에 (.venv) 표시 )
[ ] 대상 DB 확인: Select-String -Path .env -Pattern "POSTGRES_DB"
    - 검증은 리허설(gimpo365os_rehearsal)에서 수행하는 것이 기본
[ ] python manage.py check 통과
[ ] python manage.py test 통과
[ ] 점검할 서버가 떠 있음 (리허설 8001 / 운영 8000)
```

> ⛔ **STOP 조건**: 리허설 QA 중 `POSTGRES_DB`가 `gimpo365_inventory`(운영)로 확인되면 **즉시 중단**한다. QA는 화면을 눌러보는 작업이므로, 운영 DB에 붙은 상태로 진행하면 실수로 운영 데이터를 건드릴 수 있다. `.env`를 리허설로 되돌린 뒤 다시 시작한다.
> 자동 검사(check/test)가 실패하면 수동 QA로 넘어가지 않는다. 먼저 원인을 해결한다.

## 3. OS 공통 smoke QA (가장 자주 사용)

OS 틀 관련 작업(홈·sidebar·navbar·placeholder·권한 표시 등) 후 매번 확인한다.

```text
[ ] 로그인 화면이 뜨고, 정상 계정으로 로그인된다
[ ] 로그인 후 OS 홈이 표시된다 (단일 재고관리 앱이 아니라 OS 홈)
[ ] OS 홈에서 각 모듈 카드가 보인다
    - 재고관리: 사용 가능 상태
    - 공지사항 / 체크리스트 / SOP / 내부요청 / 근태: 준비 중 상태
[ ] OS 홈에서 재고관리로 이동할 수 있다
[ ] sidebar/navbar에 모듈 메뉴가 정상 표시된다
[ ] 준비 중(미구현) 모듈이 "준비 중"으로 표시된다
    - 실제 기능처럼 보이지 않는다 (입력 폼·가짜 데이터 없음)
    - 클릭 시 placeholder 화면에 "준비 중" 문구가 뜬다
[ ] 미구현 모듈이 직원/팀장/관리자 모두에게 준비 중으로 보인다
[ ] 오류 화면(500 등)이나 깨진 화면이 없다
[ ] 공통 base template 적용 후에도 화면 레이아웃이 깨지지 않는다
```

## 4. 권한별 노출 점검

권한(관리자/팀장/직원) 관련 작업을 했거나, 새 화면을 추가한 경우 확인한다.

```text
[ ] 관리자 계정: 관리자용 메뉴/기능이 보인다
[ ] 팀장 계정: 팀 단위 확인 기능이 보이고, 관리자 전용 기능은 안 보인다
[ ] 직원 계정: 본인 업무·조회 중심 화면만 보인다
[ ] 관리자 전용 등록/수정 화면에 직원 계정으로 접근 시 차단된다
[ ] (소속 결합이 있는 화면) 팀장이 자기 팀 범위만 보는지 확인
```

> 권한 매핑 방식(역할 + 소속 결합)은 OS_TECH_SPEC.md를 따른다. 새 모듈이 이 규약을 지키는지 확인.

## 5. Inventory 회귀(regression) 점검

OS 틀 작업은 원칙적으로 Inventory 핵심 로직을 건드리지 않는다. 그래도 공통 template·sidebar·URL 변경이 Inventory 화면에 영향을 줄 수 있으므로, **접근 가능 여부**만 빠르게 확인한다.

```text
[ ] Inventory 메뉴로 진입 가능
[ ] 재고현황 주요 화면 접근 가능
[ ] 입고 화면 접근 가능
[ ] 출고 화면 접근 가능
[ ] 주문/주문목록 화면 접근 가능
[ ] 관리자 리포트 화면 접근 가능
[ ] 위 화면들에서 오류 화면이 없다
```

> 이것은 "화면이 열리는지"만 보는 회귀 점검이다. 입출고 계산·거래 로직 등 **상세 동작 검증**은 `docs/modules/inventory/INVENTORY_MANUAL_QA_CHECKLIST.md`를 따른다.
> Inventory Module 코드를 수정하지 않은 OS 틀 작업에서는 상세 inventory QA 전체를 매번 하지 않는다(위 접근 확인으로 갈음).
> 단, Inventory 코드를 실제로 건드린 작업이라면 상세 QA 문서로 넘어간다.

## 6. Migration 후 점검

migration이 포함된 작업 후 확인한다. (절차 자체는 OS_DB_OPERATIONS.md)

```text
[ ] python manage.py makemigrations --check --dry-run 통과
    - 이것은 새 migration 파일을 생성하는 명령이 아니다.
    - 커밋되지 않은 모델 변경(누락된 migration)이 남아 있는지 확인만 한다.
    - 여기서 변경이 감지되면, 커밋 안 된 모델 변경이 있다는 뜻이므로 개발 단계로 되돌아간다.
[ ] 리허설 DB에서 migration이 오류 없이 적용됨
[ ] python manage.py check 통과
[ ] python manage.py test 통과
[ ] python manage.py showmigrations 로 적용 상태 확인
[ ] 새 모델/필드 관련 화면이 정상 동작
[ ] 기존 Inventory 화면이 여전히 정상 (5장 회귀 점검)
[ ] (운영 적용 시) 적용 전 운영 DB 백업 완료 — OS_DB_OPERATIONS.md
[ ] (운영 적용 시) migrate --plan 으로 적용 내용 사전 확인함
```

## 7. 운영 반영 후 점검

리허설 검증을 마치고 운영에 반영한 직후 확인한다.

```text
[ ] 운영 서버(8000) 접속 가능
[ ] 로그인 가능
[ ] OS 홈 표시
[ ] Inventory 메뉴 진입 가능
[ ] 재고현황 화면 접근 가능
[ ] 기존 데이터가 기대대로 보인다
    - 최근 거래/입출고 기록이 존재하고 정상 표시
    - 품목/공급업체 등 기준정보가 그대로 있다
    - (공지 등 다른 모듈이 있으면) 기존 데이터가 유지되어 있다
    - 화면이 뜨는 것과 데이터가 온전한 것은 별개다. 데이터 존재를 눈으로 확인한다.
[ ] 주요 메뉴 표시 정상
[ ] 준비 중 모듈이 준비 중으로 표시
[ ] 오류 화면 없음
[ ] 원내 다른 기기(모바일 등)에서 접속 가능 (같은 대역 기준)
```

> 이상 발견 시: OS_DB_OPERATIONS.md의 복구 절차 및 백업본 재복구 검토.

## 8. 새 모듈 정착 점검 (Phase 2~6, 실사용 관찰)

새 모듈을 운영 반영하고 직원 안내한 뒤, 1~2주 실사용 관찰 시 확인한다. (모듈 정착 루프)

```text
[ ] 직원이 입력을 누락하지 않는가
[ ] 직원이 메뉴 위치를 찾는가
[ ] 용어를 이해하는가
[ ] 관리자가 확인·점검하기 쉬운가
[ ] 수동 업무보다 복잡해지지 않았는가
[ ] 기록이 개인/부서 기준으로 정확히 남는가
```

> 여기서 발견된 문제는 이전 모듈로 돌아가 조정한다(정상 흐름). 조정 후 3~7장 재점검.
