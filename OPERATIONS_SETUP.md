**DB 백업/복구는 [DB_OPERATIONS.md](DB_OPERATIONS.md)** 를 참고한다.

## 1A. 원내 네트워크(LAN)에서 테스트 접속하기

> ⚠️ 이 방식은 **원내 제한 테스트용**이다. 실제 운영 배포는 별도 구성이 필요하다.
> (HTTPS, WSGI/ASGI 서버(gunicorn/uvicorn) + 리버스 프록시(nginx 등), 정적 파일 collectstatic,
>  DEBUG=False, 보안 설정. 9번 항목 참고) `runserver` 는 개발/테스트용 서버다.

### 1A.1 127.0.0.1 의 의미

```text
- 127.0.0.1 (= localhost) 는 "그 PC 자기 자신"만 가리키는 주소다.
- 따라서 python manage.py runserver (기본 127.0.0.1:8000) 로 띄우면
  서버를 띄운 그 PC 에서만 접속되고, 원내 다른 PC 에서는 접속할 수 없다.
- 다른 PC 에서 접속하려면 모든 인터페이스(0.0.0.0)로 바인딩해야 한다.
```

### 1A.2 원내 테스트 실행 명령

```text
python manage.py runserver 0.0.0.0:8000
```

- `0.0.0.0:8000` 은 서버 PC 의 모든 네트워크 인터페이스에서 8000 포트로 접속을 받는다.
- 가상환경 사용 시: `.\.venv\Scripts\python.exe manage.py runserver 0.0.0.0:8000`

### 1A.3 서버 PC 의 내부 IP 확인 (Windows)

```text
1. 명령 프롬프트(cmd) 또는 PowerShell 실행
2. ipconfig  입력
3. 현재 사용하는 어댑터(예: "이더넷" 또는 "Wi-Fi")의
   "IPv4 주소" 를 확인한다.  예) 192.168.0.25
```

(PowerShell 대안: `ipconfig | findstr IPv4`)

### 1A.4 다른 PC 에서 접속하는 주소

```text
http://<서버PC_IP>:8000/
예) http://192.168.0.25:8000/
```

- 접속하는 PC 와 서버 PC 가 **같은 원내 네트워크(같은 공유기/스위치)** 에 있어야 한다.

### 1A.5 ALLOWED_HOSTS 에 서버 IP 허용

Django 는 `ALLOWED_HOSTS` 에 없는 호스트로 접속하면 차단(400)한다. `.env` 의
`DJANGO_ALLOWED_HOSTS` 에 **서버 PC 의 내부 IP** 를 추가한다.

```text
# .env  (쉼표로 구분)
DJANGO_ALLOWED_HOSTS=localhost,127.0.0.1,192.168.0.25
```

- IP 가 바뀌면(공유기 재시작 등) 값도 갱신한다. 고정이 필요하면 공유기에서 고정 IP 할당.
- `.env` 의 `DJANGO_DEBUG=True` 인 테스트 환경 기준이다.

### 1A.6 Windows 방화벽에서 8000 포트 허용

처음 `runserver 0.0.0.0:8000` 실행 시 Windows 방화벽 경고가 뜨면 **개인 네트워크 허용**을
체크한다. 수동 설정이 필요하면(관리자 권한 PowerShell):

```text
netsh advfirewall firewall add rule name="gimpo365 inventory 8000" dir=in action=allow protocol=TCP localport=8000
```

- 회사/도메인 네트워크 환경이면 해당 프로필도 허용이 필요할 수 있다.
- 테스트 종료 후 규칙을 제거하려면:
  `netsh advfirewall firewall delete rule name="gimpo365 inventory 8000"`

### 1A.7 접속이 안 될 때 점검

```text
- 서버 PC 와 클라이언트 PC 가 같은 네트워크인지
- runserver 가 0.0.0.0:8000 으로 떠 있는지 (127.0.0.1 이 아님)
- .env 의 DJANGO_ALLOWED_HOSTS 에 서버 IP 가 포함됐는지 (400 이면 이 항목)
- Windows 방화벽에서 8000 포트가 허용됐는지 (접속 자체가 안 되면 이 항목)
- 백신/엔드포인트 보안 SW 가 포트를 막고 있지 않은지
```

## 1B. 알파테스트 데이터 초기화 (DEBUG 전용)

> ⚠️ **운영 환경에서 사용 금지.** 이 절차는 알파테스트 데이터를 비우기 위한 **비운영 teardown**
> 이다. `DEBUG=False`(운영) 환경에서는 `reset_alpha_data` 명령이 **무조건 거부**된다.
> 거래 기록은 운영에서 **삭제하지 않는다**(오입력은 취소/철회). 이 명령은 그 원칙과 별개의
> 테스트 전용 도구다.

### 왜 Admin 에서 StockTransaction delete 를 열지 않는가

```text
- 재고 거래(원장)는 운영에서 물리 삭제하지 않는다. 오입력은 "취소(CANCELED)" 이력으로 남긴다.
  (PRODUCT_SPEC §6.1 / TECH_SPEC §0) → Admin add/delete 는 계속 닫아 둔다.
- 알파테스트 데이터 정리는 운영 경로가 아니라, DEBUG 전용 management command 로만 수행한다.
- 이 명령은 PROTECT FK 를 깨지 않고(자식→부모 순서로 삭제), service/Admin/모델을 변경하지 않는다.
```

### reset_alpha_data 명령

기본 동작: **StockTransaction → ManagedItem → Item → Supplier 삭제.**
유지: **Department(전체), User(전체), superuser/ADMIN.** (사용자/부서 구조는 그대로)

```text
# 삭제 전 미리보기 (실제 삭제 없음)
python manage.py reset_alpha_data --dry-run

# 재고 데이터 초기화 (확인 프롬프트: RESET 입력)
python manage.py reset_alpha_data

# 확인 프롬프트 생략
python manage.py reset_alpha_data --yes

# 테스트 사용자까지 삭제 (username 이 test_ 로 시작하거나 _test 로 끝나는 계정만,
# superuser / ADMIN 은 어떤 경우에도 삭제하지 않음)
python manage.py reset_alpha_data --yes --delete-test-users
```

가드:

```text
- DEBUG=False → 즉시 CommandError 로 중단 (운영 보호)
- --yes 없으면 'RESET' 입력 확인 요구
- --dry-run 은 삭제 대상/건수만 출력
- 실행 후 삭제 건수 요약 출력
```

### reset_training_transactions — 알파/교육용 거래기록만 초기화 (DEBUG 전용)

> ⚠️ **운영 환경 사용 금지.** 알파테스트 또는 **직원 교육 중 입력된 거래기록(StockTransaction)** 을
> 정식 운영 전에 정리하기 위한 비운영 도구다. **실제 운영 중에는 사용하지 않는다.**
> 운영 화면에는 초기화 버튼을 두지 않으며, 이 명령은 셸에서만 실행한다.

`reset_alpha_data` 와 달리 **공급업체/품목/관리품목/부서/사용자는 모두 유지**하고
**StockTransaction 만 삭제**한다. (마스터·조직 데이터는 그대로)

```text
# 삭제 대상 미리보기 (기본 동작 = dry-run, 실제 삭제 없음)
python manage.py reset_training_transactions

# 실제 삭제 (--yes 필요)
python manage.py reset_training_transactions --yes

# 거래일자(occurred_at) 기간 필터 — 시작/종료일 포함
python manage.py reset_training_transactions --yes --from 2026-06-01 --to 2026-06-25
```

출력:

```text
- 삭제 대상 총 건수
- 거래유형별 건수: INITIAL_COUNT / IN / OUT(사용·폐기·분실·증정·기타출고 합계) / ADJUSTMENT (기타 유형이 있으면 함께)
- 상태별 건수: PENDING / APPROVED / REJECTED / CANCELED (기타 상태가 있으면 함께)
```

가드:

```text
- DEBUG=False → 즉시 CommandError 로 중단 (운영 보호)
- 기본은 dry-run, --yes 가 있을 때만 실제 삭제
- StockTransaction 만 삭제하며 거래 status 직접 변경/신규 생성은 하지 않는다 (삭제만)
- 기간 필터 기준은 거래일자(occurred_at)
```

### 전체 DB 초기화가 필요한 경우 (대안)

품목/거래 ID 까지 1 부터 새로 시작하거나 스키마째 비우려면:

```text
# (A) 모든 데이터 비우기(시퀀스 리셋 포함). superuser 도 삭제되므로 이후 재생성 필요.
python manage.py flush

# (B) DB 를 통째로 다시 만들기 (가장 깨끗)
#   PostgreSQL 에서 DB drop & create 후
python manage.py migrate
```

### 초기화 후 확인

```text
- reset_alpha_data 는 ADMIN/부서를 유지하므로 보통 추가 작업이 필요 없다.
- flush / drop-create 를 했다면 createsuperuser 로 ADMIN 을 다시 만든다.
- 어느 경우든 활성 ADMIN 계정이 최소 2개인지 확인한다. (PRODUCT_SPEC §14.1)
```

### seed_alpha_inventory — 알파테스트 기본 데이터 생성 (DEBUG 전용)

> ⚠️ **운영 환경 사용 금지.** 실제 운영 데이터 세팅용이 아니라 **알파테스트용 샘플 데이터**다.
> 부서/테스트 사용자/공급업체/품목/관리품목/초기재고를 한 번에 만든다.
> 초기재고는 **기존 service(`request_initial_count`)** 로 생성되어 APPROVED 로 현재고에 반영된다.
> 테스트 데이터는 username `_test` 접미, 공급업체/품목명 `[테스트]` prefix 로 표시되어
> `reset_alpha_data --delete-test-users` 와 호환된다.

```text
# 생성 예정 미리보기 (실제 생성 없음)
python manage.py seed_alpha_inventory --dry-run

# 전체(피부실+치료실) 생성 (확인: SEED 입력)
python manage.py seed_alpha_inventory

# 확인 생략
python manage.py seed_alpha_inventory --yes

# 특정 부서만
python manage.py seed_alpha_inventory --department skin
python manage.py seed_alpha_inventory --department treatment
python manage.py seed_alpha_inventory --department all

# 입고/출고 샘플 거래(당일)도 생성 (service 사용, [seed] 메모로 idempotent)
python manage.py seed_alpha_inventory --yes --with-transactions

# 과거 거래이력까지 생성 (기간 필터/과거 취소 제한 확인용)
python manage.py seed_alpha_inventory --yes --with-transactions --with-history
```

`--with-history` 설명:

```text
- 과거 거래(3/10/35/100일 전 입·출고)를 service 로 생성한 뒤,
  seed 거래의 created_at(입력일시)만 과거값으로 보정한다. (메모 [seed-history])
- 목적: 오늘/최근 7일/이번 달/최근 3개월 기간 필터, 수동 기간 조회,
  STAFF 과거 입력 거래 직접 취소 불가(입력일시 기준 당일)를 수동 확인.
- idempotent: 재실행해도 [seed-history] 메모로 중복 생성하지 않는다.
- 운영 데이터 생성용이 아니라 DEBUG 전용 알파테스트 데이터다.
  (created_at 시간 보정은 seed command 내부의 테스트 전용 예외이며, 운영 로직에서는 하지 않는다.)
```

특징:

```text
- idempotent: 여러 번 실행해도 중복 생성되지 않음(get_or_create + 초기재고 유일성 확인).
- 생성/재사용 건수 요약 출력.
- DEBUG=False 면 CommandError 로 즉시 중단.
- 테스트 계정 공통 비밀번호: test1234!
- 생성 계정: manager_test(MANAGER), skin_staff_test/skin_leader_test(피부실),
  treatment_staff_test/treatment_leader_test(치료실). superuser/ADMIN 은 만들지 않는다.
- 일부 품목은 일부러 "최소재고 이하"가 되도록 구성(부족 품목 화면 테스트용).
```

### 알파테스트 1회 세팅 흐름 (reset + seed)

```text
1) python manage.py reset_alpha_data --dry-run     # 무엇이 지워질지 확인
2) python manage.py reset_alpha_data               # 재고 데이터 초기화
3) python manage.py seed_alpha_inventory --dry-run # 무엇이 생길지 확인
4) python manage.py seed_alpha_inventory           # 샘플 데이터 생성
5) python manage.py runserver 0.0.0.0:8000         # (원내 LAN 테스트는 §1A)
6) STAFF/MANAGER 테스트 계정으로 로그인해 알파테스트
   - 예: skin_staff_test / manager_test  (비밀번호 test1234!)
```

> 위 명령들은 모두 **알파테스트용**이며 운영 데이터 세팅용이 아니다.

## 2. 관리자(ADMIN) 계정 생성 — 최소 2개

역할 변경과 ADMIN 권한 부여는 ADMIN 만 할 수 있다. ADMIN 이 1명뿐이고 접근이 막히면
운영 복구가 어려우므로 **활성 ADMIN 계정을 최소 2개** 둔다. (PRODUCT_SPEC §14.1)

```text
python manage.py createsuperuser
```

- `createsuperuser` 로 만든 계정은 자동으로 **role=ADMIN, is_staff=True, is_superuser=True** 가 된다.
- 위 명령을 **2회 이상** 실행해 ADMIN 을 2명 이상 만든다.
- ADMIN 은 `/admin/` (Django Admin)과 일반 운영 화면 모두 사용할 수 있다.

## 3. 부서 / 사용자 등록 (Django Admin)

ADMIN 으로 `/admin/` 에 로그인해 등록한다.

### 3.1 부서

```text
- 피부실(스킨앤라인) : is_active=True, active_for_inventory=True
- 치료실             : is_active=True, active_for_inventory=True
- (탕전실은 v0.1 재고관리 대상 아님: active_for_inventory=False)
```

### 3.2 사용자(직원 계정)

```text
- 직원별 개별 계정 생성 (공유 계정 사용 금지)
- 각 사용자에 역할(STAFF / TEAM_LEADER / MANAGER / ADMIN)과 부서 지정
- 승인 가능 계정(MANAGER 이상)을 최소 2개 둔다 (승인 병목 방지)
- 퇴사/비활성 직원은 삭제하지 않고 is_active=False 로 비활성화
```

**개인별 계정 원칙 (중요):**

```text
- 공유 계정(예: 치료실1, 데스크1) 사용을 금지한다.
- 이유: 입고/출고/취소/승인의 created_by / approved_by / canceled_by 추적성 유지.
- 직원은 본인 계정으로만 로그인한다.
```

**사용자 생성 후 비밀번호 설정 (필수):**

```text
- Django Admin 의 "사용자 추가" 화면에서 username + 비밀번호(2회 확인)를 입력해 생성한다.
  → 이 흐름을 따르면 비밀번호 없는 계정이 생기지 않는다.
- 생성 직후 "변경" 화면에서 이름/역할/부서를 지정한다.
- 비밀번호를 잊었거나 미설정인 계정은 다음으로 재설정한다:
    Admin 사용자 변경 화면의 비밀번호 변경 링크, 또는
    python manage.py changepassword <username>
```

**직원 비밀번호 변경 (셀프 서비스):**

```text
- 직원은 로그인 후 상단 네비게이션의 "비밀번호 변경"(/accounts/password-change/)에서
  본인 비밀번호를 직접 변경할 수 있다.
- 비활성(is_active=False) 사용자는 로그인할 수 없다.
```

역할 요약:

```text
STAFF        : 본인 부서 입고/출고/조회/실사조정 요청/초기재고 요청/당일 본인 거래 취소
TEAM_LEADER  : STAFF + 본인 부서 전체 거래 조회/당일 부서 거래 취소
MANAGER      : 전체 부서 조회·입력 + 승인/반려/취소 + 초기재고 즉시 승인
ADMIN        : 전체 + 사용자/역할 관리 + Django Admin
```

## 4. 마스터 데이터 등록 (Django Admin)

```text
1) 공급업체(Supplier) 등록
2) 품목 마스터(Item) 등록  — 분류는 고정 선택값, 구분 정보는 품목명에 포함 (예: 거즈 5x5)
3) 부서별 관리품목(ManagedItem) 등록 — Item + 부서 + 관리단위 + 최소재고 (+ 기본 공급업체)
```

> 주의: 관리단위(unit)는 운영 개시(승인 거래 발생) 후에는 변경할 수 없다.
> 단위를 바꿔야 하면 기존 관리품목을 비활성화하고 새로 만든다. (PRODUCT_SPEC §9.4)

## 5. 초기재고 입력 및 승인 (하드리셋 핵심)

온보딩 순서 (PRODUCT_SPEC §11.3):

```text
1. 부서 생성            (3.1)
2. 품목 마스터 생성      (4)
3. 부서별 관리품목 생성   (4)
4. 실제 재고 실사        (현장)
5. 초기재고(INITIAL_COUNT) 입력
6. MANAGER 또는 ADMIN 승인
7. 초기재고 승인 완료 확인
8. 입고/출고 운영 시작
```

> v0.1.1 흐름 변경: 별도 '초기재고 입력' 화면은 **'실사조정 요청'으로 통합**되었다.
> 선택 품목에 승인된 최초 재고가 없으면 '최초 재고 입력'(INITIAL_COUNT)으로, 있으면
> '실사조정'(ADJUSTMENT)으로 자동 처리된다. **요청은 TEAM_LEADER 이상**만 가능하다.

- TEAM_LEADER 가 입력한 최초 재고는 **승인 대기(PENDING)** 상태다.
- MANAGER/ADMIN 이 입력하면 즉시 승인된다. (STAFF 는 최초 재고 입력/실사조정 요청 불가)
- 승인 대기 큐에서 **일괄 승인**으로 여러 초기재고를 한 번에 처리할 수 있다.
- **초기재고 승인 전에는 해당 품목의 입고/출고 운영을 시작하지 않는다.**
- 승인되지 않은 품목은 현재고가 0 으로 보일 수 있으므로 운영 시작 전 승인 여부를 반드시 확인한다.

### 5.1 최초재고 승인 전 입고/출고 불가 (시스템 강제)

최초재고는 재고관리의 기준점이므로, **승인된 최초재고(INITIAL_COUNT)가 없는 관리품목에는
입고/출고/일반 실사조정이 먼저 등록되지 않도록 시스템이 차단**한다. (운영 규칙이 아닌 강제 규칙)

| 품목 상태 | 입고/출고 | 실사조정 요청 | 최초재고 요청 |
| --- | --- | --- | --- |
| 승인된 최초재고 **없음** | ❌ 차단 | 자동으로 **최초재고 입력**으로 처리 | ✅ 가능 |
| 최초재고 **승인 대기(PENDING)** | ❌ 차단 | (위와 동일) | ❌ 중복 차단 |
| 승인된 최초재고 **있음** (수량 0 포함) | ✅ 가능 | 일반 ADJUSTMENT 로 처리 | ❌ 중복 차단 |

차단 시 표시되는 안내 메시지:

```text
- 최초재고 미등록:
  "이 품목은 최초 재고 등록이 승인되지 않아 입고/출고 등록을 할 수 없습니다.
   먼저 최초 재고 등록을 진행해주세요."
- 최초재고 승인대기:
  "이 품목은 최초 재고 등록이 승인 대기 중입니다.
   승인 완료 후 입고/출고 등록이 가능합니다."
```

> 실물 재고가 0 인 품목이라도 **최초재고를 수량 0 으로 입력·승인**하면 입고/출고가 가능해진다.
> 즉, "재고가 없어서" 입고가 막히는 것이 아니라 "기준점(최초재고)이 승인되지 않아서" 막히는 것이다.

## 6. 운영 시작 후 일상 사용 (요약)

```text
- 입고: 입고 등록 화면 (즉시 반영)
- 출고: 출고 등록 화면 (사용/폐기/분실/증정/기타출고 구분, 현재고 초과 불가)
- 현재고/최소재고 이하 품목: 조회 화면
- 최초 재고 입력 / 재고 불일치: 실사조정 요청(TEAM_LEADER 이상) → MANAGER 승인
- 오입력: 삭제가 아니라 "취소"로 처리. 직접 취소는 입력일시 기준 당일까지(STAFF 본인 / TL 부서, MANAGER 이상 전체)
```

팀장 외 1명 이상이 위 업무(입고/출고/조회/오입력 취소)를 익혀
특정 개인에게 지식이 묶이지 않게 한다. (PRODUCT_SPEC §11.5)

## 6A. 운영 기준 (v0.1.2 알파 피드백 반영)

5일간 알파테스트에서 나온 피드백을 정리한 운영 기준이다. v0.2 고도화가 아니라 정식 운영 전 안정화 기준이다.

### 6A.1 거래 상세 / 메모 확인

```text
- 거래이력 목록의 각 거래에서 "상세" 링크로 거래 상세 화면을 연다.
- 상세 화면에서 거래일자·입력일시·거래유형·품목·부서·수량/증감·공급업체·상태·메모·
  입력자·승인자·취소자/취소일시/취소사유 등을 한눈에 확인한다.
- 메모는 입고/출고/실사조정 시 사유·특이사항을 남기는 용도다. (예: 로트번호, 교환 사유)
- 상세 접근 권한은 거래이력과 동일하다. STAFF/TEAM_LEADER 는 본인 부서 거래만,
  MANAGER/ADMIN 은 전체. (권한 범위는 넓어지지 않는다.)
```

### 6A.2 입고 등록 오류 처리 (취소 후 재등록)

```text
- 입고 등록 후 품목·수량·공급업체·입고일자 중 하나라도 잘못 입력했으면,
  그 입고 거래를 "취소"하고 올바른 내용으로 다시 등록한다.
- 운영 중 거래 기록을 직접 수정하지 않는다. (수정 기능 없음 — 의도된 정책)
- 취소 기록(취소자/취소일시/취소사유)을 남겨 이력을 추적할 수 있게 한다.
```

### 6A.3 공급업체 입력 기준

```text
- 공급업체는 입고처 추적·리포트를 위한 필수 정보다. 입고 시 공급업체를 확인한다.
- 입고 화면에서 선택 품목에 기본 공급업체가 있으면 공급업체 입력값이 그 값으로 자동 선택된다.
  (기본 공급업체는 "초기값"일 뿐 고정이 아니다.)
- 동일 품목이 여러 공급업체에서 들어올 수 있으므로, 필요하면 등록된 다른 공급업체로 변경한다.
- 공급업체는 등록된 공급업체 목록에서만 선택한다. 목록에 없는 이름을 자유 텍스트로 저장할 수 없다.
- 잘못된 공급업체로 등록했다면 6A.2 와 같이 취소 후 재등록한다.
```

### 6A.4 입고 / 출고 / 실사조정 구분 (다이어트약 교환 등)

```text
- 입고: 구입해서 새로 들어온 물품.
- 출고: 환자에게 나가거나, 사용할 수 없는 상태가 되어 실제 재고에서 빠지는 경우.
- 실사조정: 교환으로 기존 처방이 회수되는 등 실제 재고 기준을 맞춰야 하는 경우.
- 특이사항(교환·처방 단계 변경 등)은 거래 메모에 사유를 남긴다.
- 참고: 반품/교환/폐기 전용 거래유형은 향후 검토 대상이며, 현재는 새 거래유형을 추가하지 않는다.
  (위 입고/출고/실사조정 + 메모로 처리한다.)
```

### 6A.5 품목 삭제가 아니라 "사용중지"로 관리

```text
- 거래이력이 있는 품목/관리품목은 삭제하지 않는다. (이력 보존)
- 더 이상 쓰지 않는 품목은 관리자 판단으로 "사용중지(비활성화)" 처리한다.
  (관리품목 is_active=False → 입고/출고 화면 후보에서 제외된다.)
- 현재는 별도의 "삭제 요청" 기능을 제공하지 않는다.
- 향후 현재고 0 + 최근 N일간 입출고 없음 등을 기준으로 관리자에게
  "장기 미사용 품목 확인 필요" 알림을 제공하는 방식을 검토한다.
```

### 6A.6 알파/교육용 거래기록 초기화

```text
- 알파테스트/직원 교육 중 입력된 거래기록은 reset_training_transactions 로 정리한다. (§1B 참조)
- 이 명령은 알파/교육용이며 실제 운영 중에는 사용하지 않는다. DEBUG=False 에서는 실행되지 않는다.
```

## 6B. 주문 장바구니 / 주문 기록 (v0.2.0)

입고/출고 이전 단계인 "주문 여부"를 기록해 중복 주문과 주문 누락을 방지한다.
로테이션 근무 환경에서 누가 언제 무엇을 주문했는지 추적하기 위한 기능이다.

### 6B.1 핵심 원칙

```text
- 주문은 현재고를 변경하지 않는다. (장바구니 담기/주문 확정/상태 변경 모두 현재고 무관)
- 실제 재고 증가는 기존 "입고 등록"으로만 처리한다. (주문과 입고는 별개)
- 주문(Order)과 거래원장(StockTransaction)은 분리되어 있다.
```

### 6B.2 장바구니 → 주문 확정 흐름

```text
1. 재고현황 또는 "최소재고 이하 품목"에서 품목의 [담기] 로 주문 장바구니에 담는다.
   - 주문수량을 입력하고(기본 1) 공급업체를 선택한다.
   - 같은 (품목 + 공급업체) 조합을 다시 담으면 새 행이 아니라 수량이 합산된다.
2. 좌측 퀵메뉴 "주문 장바구니"에서 수량/공급업체/메모를 수정하거나 항목을 삭제한다.
3. "주문 확정으로" → 확인 화면에서 주문일자/외부 주문번호(선택)/메모를 입력하고 확정한다.
4. 확정 시 공급업체별로 주문이 나뉘어 생성된다.
   - 예: 장바구니에 A업체 품목 2개 + B업체 품목 1개 → Order 2건 생성.
5. 확정 후 장바구니는 비워지며, 생성된 주문 상태는 "주문완료(ORDERED)"다.
```

### 6B.3 공급업체 / 주문번호

```text
- 공급업체는 입고처 추적·리포트를 위한 정보다. 담을 때 품목 기본 공급업체가 초기값으로 잡히며,
  등록된 다른 공급업체로 변경할 수 있다. (등록 공급업체만 선택 가능 — 자유 텍스트 불가)
- 외부 주문번호(쇼핑몰/거래처 번호)는 선택 입력이다. 전화주문 등 외부번호가 없으면 비워둔다.
- 내부 주문번호는 YYMMDD-순번 형식으로 자동 생성된다. (예: 260701-1, 260701-2)
  외부번호가 없어도 내부 주문번호로 주문을 관리한다.
```

### 6B.4 입고완료 / 취소

```text
- 주문 물품이 도착하면: 기존 "입고 등록"으로 실제 재고를 올린 뒤, 주문을 "입고완료(RECEIVED)"로 바꾼다.
  - 주문 상세의 "입고 등록으로 이동"은 품목/공급업체를 입고 화면 초기값으로 넘겨준다(편의 기능).
  - 주문을 RECEIVED 로 바꿔도 현재고는 변하지 않는다. (재고는 입고 등록에서만 증가)
- 잘못 주문한 경우: 주문완료(ORDERED) 상태에서 취소한다. 취소해도 현재고에는 영향이 없다.
  - 입고완료(RECEIVED) 주문은 취소할 수 없다.
- 주문 취소·입고완료 처리는 주문자 본인 또는 MANAGER/ADMIN 이 할 수 있다.
```

### 6B.5 권한 / 조회 범위

```text
- 주문 장바구니 담기·주문 생성: STAFF/TEAM_LEADER 이상 가능(접근 가능한 부서 품목 한정).
- 주문 조회: STAFF/TEAM_LEADER 는 본인 또는 본인 부서(주문자 기준) 주문, MANAGER/ADMIN 은 전체.
- 대시보드 "미입고 주문"과 좌측 퀵메뉴 "주문 장바구니 / 주문 목록"에서 빠르게 접근한다.
```

## 6C. 주문서 기반 입고등록 (v0.2.1)

주문서(입고 예정 기록)와 입고등록(실제 재고 증가)을 연결한다. 주문서에서 아는 값은 자동으로 채우고,
실제 입고 시 확인해야 하는 값은 직원이 입력한다.

### 6C.1 원칙

```text
주문서       = 입고 예정 기록 (현재고 변경 없음)
입고등록      = 실제 재고 증가 기록 (APPROVED 입고 StockTransaction 생성)
현재고        = APPROVED StockTransaction.quantity_delta 합계 (변함없음)
- 주문 확정/주문 상태 변경만으로는 현재고가 절대 바뀌지 않는다.
- 실제 재고 증가는 주문 상세의 [입고등록] 으로 만든 입고 거래로만 발생한다.
```

### 6C.2 입고등록 절차 (OrderItem 단위)

```text
1. 좌측 퀵메뉴 "주문 목록" 또는 "입고대기 품목"에서 주문/품목을 연다.
2. 주문 상세에서 품목별로 기입고/잔여 수량과 상태(미입고/부분입고/입고완료)를 확인한다.
3. 잔여수량이 있는 품목의 [입고등록] 폼에 입고수량·입고일자·단가·유통기한·메모를 입력하고 등록한다.
   - 관리품목/공급업체/주문수량/주문번호는 주문서에서 자동으로 연결된다.
   - 입고수량 기본값은 잔여수량이다. 나눠서 여러 번 입고(부분입고)할 수 있다.
4. 등록하면 실제 입고 거래가 생성되어 현재고가 증가하고, 주문 상태가 자동 갱신된다.
   (기입고 0 → 주문완료 / 일부 입고 → 부분입고 / 잔여 0 → 입고완료)
```

버튼명은 '입고완료'가 아니라 '입고등록'이다. (상태만 바꾸는 것이 아니라 실제 입고 거래를 만든다는 의미)
기존 주문 단위 '입고완료' 버튼은 제거되었다. 입고는 반드시 품목 단위로 등록한다.

### 6C.3 주문수량 초과 입고 차단

```text
- 주문서 기반 입고등록은 주문 잔여수량까지만 허용한다.
- 잔여수량보다 많이 등록하려 하면 차단되며, 입고 거래가 생성되지 않는다.
- 추가증정/덤/추가입고 등 초과분은 일반 "입고 등록" 화면에서 별도로 처리하고,
  메모에 '추가증정' 등 사유를 남긴다.
```

### 6C.4 단가 · 유통기한 필수 (입고 데이터 품질)

```text
- 입고등록 시 단가는 필수이며 0보다 커야 한다. (일반 입고/주문서 기반 입고 모두)
- 유통기한도 필수다. 표시가 없는 품목은 [유통기한 없음]을 선택한다.
  선택 시 입고일로부터 3년 뒤 날짜가 자동 입력된다. (빈 값으로 저장하지 않는다)
- 3년 이상 출고되지 않은 물품은 장기 미사용/재고 점검 대상으로 확인한다.
```

### 6C.5 잔여취소 미구현 / 취소

```text
- 이번 버전에서는 '잔여수량 취소' 기능을 만들지 않는다. 미입고 잔여수량은 계속 남아 보인다.
  공급업체 품절/미도착 등은 주문 메모로 남긴다.
- 잘못 등록한 입고 거래는 거래이력에서 취소한다. 취소하면 기입고 수량에서 자동 제외되어
  잔여수량이 다시 늘고, 재입고할 수 있다.
- 주문 자체가 잘못된 경우 주문완료/부분입고 상태에서 주문을 취소한다. (현재고 영향 없음)
```

```text
     python manage.py changepassword <username>
6) 복구 후, 활성 ADMIN 이 다시 2개 이상인지 확인한다.
```

운영 원칙:


## 6F. 입고/출고/실사조정 부서 필터 (v0.2.2 후속)

MANAGER/ADMIN 은 전 부서 관리품목이 한 목록에 나와 선택이 번거로울 수 있다. 이를 위해
입고등록/출고등록/실사조정 화면에 **부서 필터**를 추가했다.

```text
- MANAGER/ADMIN: 화면 상단에 '부서 필터' 선택이 보인다. 부서를 고르면 관리품목 자동완성
  후보가 해당 부서 품목만 표시된다. (다른 부서 품목을 골랐다가 부서를 바꾸면 선택이 초기화된다)
- STAFF/TEAM_LEADER: 부서 필터가 보이지 않으며 기존과 동일하게 본인 부서 품목만 다룬다.
- 부서 필터는 화면에서 후보를 좁혀주는 편의 기능일 뿐이다. 서버가 허용하는 관리품목 범위는
  기존 권한 범위 그대로이며, 부서 필터로 권한이 넓어지지 않는다.
```

## 6G. 장바구니 선택 주문 · 숫자 콤마 · 미입고 잔여마감 (v0.2.2 후속)

### 6G.1 장바구니에서 선택 항목만 주문
```text
- 장바구니 각 항목에 체크박스가 있다. [전체선택]으로 한 번에 선택할 수 있다.
- [선택 항목 주문하기]: 체크한 항목만 공급업체별 주문으로 확정한다.
  선택하지 않은 항목은 장바구니에 그대로 남는다. (일부만 주문 가능)
- [전체 항목 주문하기]: 기존처럼 장바구니 전체를 주문한다.
- 선택 0개이거나 공급업체 미지정 항목이 있으면 확정되지 않는다.
- 본인 장바구니 항목만 대상이며 권한 범위는 넓어지지 않는다.
```

### 6G.2 숫자 1,000 단위 콤마
```text
- 현재고/수량/단가/금액 등 화면 표시 숫자에 천 단위 콤마가 붙는다. (예: 1,000 · 12,000)
- 입력칸(수량/단가 등)에는 콤마가 들어가지 않는다. (브라우저 숫자 입력 오류 방지)
```

### 6G.3 미입고 잔여마감
```text
- 부분입고 후 더 이상 입고되지 않을 잔여수량은 [미입고 잔여마감]으로 정리한다.
  (품절/환불/타업체 재주문/주문 취소/입고 포기/기타 사유 선택)
- 잔여마감은 재고 증감이 아니다. 입고 거래를 만들지 않으며 현재고를 바꾸지 않는다.
- 미처리잔여 = 주문수량 - 기입고 - 잔여마감. 미처리잔여가 0이 되면 입고대기 목록에서 사라진다.
- 마감수량은 미처리잔여 이하여야 하고 사유는 필수다. 취소된 주문/입고완료 품목은 마감할 수 없다.
- 잔여마감 내역(수량/사유/처리자)은 주문 상세에 표시된다.
- 잘못 등록한 입고 거래를 취소하면 기입고가 줄어 미처리잔여가 다시 생기고 재입고할 수 있다.
```

```text
  파일명 예: inventory_stock_snapshot_YYYYMMDD.xlsx, inventory_monthly_report_YYYYMMDD.xlsx
```

## 6I. 운영 시작 전 초기화 · 기준정보 점검 (v0.2.5)

### 6I.1 reset_operational_data (터미널 전용)
직원교육/알파테스트 후 **정식 운영 시작 전**, 기준정보는 유지하고 운영기록만 지운다.

```text
python manage.py reset_operational_data            # 기본 dry-run (삭제 예정 건수만 출력)
python manage.py reset_operational_data --yes      # 실제 삭제
python manage.py reset_operational_data --yes --allow-production   # DEBUG=False 운영환경에서 강제
```
```text
- 삭제: StockTransaction / CartItem / OrderItem / Order (운영기록)
- 유지: User / Department / Supplier / Item / ManagedItem / OrderCart(껍데기) 등 기준정보
- 안전장치: 기본 dry-run, --yes 에서만 삭제, transaction.atomic,
  DEBUG=False 에서는 --allow-production 없이는 차단.
- 주의: StockTransaction 을 지우면 현재고 기준(APPROVED 합계)도 사라진다.
  초기화 후 각 관리품목별 최초재고 입력/승인을 다시 진행해야 한다.
```
