## 1. 문서 목적

이 문서는 김포365한의원의 내부 운영 시스템인 `김포365OS`의 제품 목적, 사용자 범위, 핵심 기능, 모듈 구성, 운영 원칙, 개발 우선순위를 정의한다.

김포365OS는 단순한 재고관리 프로그램이 아니다.
기존 `gimpo365-inventory` MVP를 첫 번째 모듈로 포함하고, 향후 공지사항, 체크리스트, SOP, 직원교육, 내부요청, 결재, 근태/근무표 등 김포365한의원의 내부 운영 업무를 단계적으로 통합하기 위한 시스템이다.

이 문서는 Claude Code 및 향후 개발자가 김포365OS의 방향을 이해하고, 기능을 추가할 때 기존 운영 철학과 충돌하지 않도록 하기 위한 기준 문서로 사용한다.

## 2. 프로젝트 정의

## 2.1 프로젝트명

```text
김포365OS
gimpo365-os
```

## 2.2 한 문장 정의

김포365OS는 김포365한의원의 모든 상시 근무자가 매일 접속하여 공지, 체크리스트, 재고관리, 업무 매뉴얼, 교육자료, 내부 요청을 확인하고 처리하는 내부 업무 허브이자, 관리자가 병원 운영 상태를 점검하는 운영 관리 시스템이다.

## 2.3 제품 성격

김포365OS는 다음 성격을 가진다.

```text
1. 전 직원 업무 허브
2. 관리자 운영 점검 시스템
3. 업무 참조 공간
4. 직원 교육자료 허브
5. 김포365한의원 내부 인트라넷
6. 모듈형 병원 운영 시스템
```

다만 초기 버전에서 모든 기능을 구현하지 않는다.
현재는 기존 `gimpo365-inventory`를 기반으로 김포365OS의 최소 틀을 만들고, 이후 필요한 모듈을 단계적으로 추가한다.

## 3. 프로젝트 배경

김포365한의원은 다음과 같은 운영 특성을 가진다.

```text
전 직원이 기본 업무에 숙달되어 있음
일용직 근무자를 제외한 모든 근무자가 장기 근속 중임
재고관리, 공지, 체크리스트, 업무 매뉴얼, 교육자료가 여러 도구에 흩어질 가능성이 있음
관리자는 직원 업무 수행 여부와 운영 상태를 주기적으로 점검해야 함
신규 직원 교육 및 인수인계 시 기준 자료가 필요함
```

기존에는 업무가 다음 방식으로 분산될 수 있었다.

```text
구두 전달
카카오톡 공지
엑셀
종이 체크리스트
개별 문서
개별 PC 파일
직원 기억에 의존한 인수인계
```

이 방식은 초기에는 간단하지만, 장기적으로 다음 문제가 생긴다.

```text
공지 누락
업무 기준 불명확
신규 직원 교육 편차
체크리스트 수행 여부 확인 어려움
업무 요청 처리상태 추적 어려움
관리자가 운영 상태를 한눈에 보기 어려움
문서 최신본 확인 어려움
```

김포365OS는 이런 문제를 해결하기 위해 만든다.

## 4. 제품 목표

## 4.1 장기 목표

김포365OS의 장기 목표는 다음과 같다.

```text
김포365한의원의 내부 운영 업무를 하나의 시스템 안에서 확인, 수행, 기록, 점검할 수 있게 한다.
```

## 4.2 초기 목표

초기 목표는 다음과 같다.

```text
기존 Inventory Module을 김포365OS의 첫 번째 모듈로 포함한다.
로그인 후 OS 홈에서 모듈을 선택하게 한다.
관리자에게 향후 확장될 모듈 구조를 보여준다.
전 직원이 매일 접속할 수 있는 기본 업무 허브 형태를 만든다.
공지사항, 체크리스트, SOP 등 후속 모듈을 추가할 수 있는 구조를 만든다.
```

## 4.3 이번 단계의 목표

이번 단계의 목표는 기능을 많이 추가하는 것이 아니다.

이번 단계의 목표는 다음과 같다.

```text
김포365OS라는 상위 시스템의 틀을 만든다.
Inventory Module을 Module 1로 배치한다.
OS 홈과 공통 메뉴 구조를 만든다.
미구현 모듈은 준비 중 상태로 표시한다.
기존 Inventory Module 기능은 최대한 건드리지 않는다.
```

재고관리는 파트별 마감담당자가 매일 수행하는 필수 프로세스이다.
따라서 1단계에서 Inventory Module만 제공되더라도 데스크, 피부관리실, 치료실 등 전 파트에 매일 접속할 실질적 동인이 존재한다.

김포365OS의 초기 일일 접속 동인은 단순한 기대가 아니라, 이미 운영 중인 파트별 재고관리 마감 프로세스를 시스템화하는 데서 발생한다.

## 5. 사용자 범위

## 5.1 사용 대상

김포365OS는 일용직 근무자를 제외한 모든 상시 근무자가 사용하는 것을 목표로 한다.

초기부터 전체 상시 근무자가 사용할 수 있어야 한다.

사용자 후보는 다음과 같다.

```text
대표원장
총괄실장
진료원장
데스크 팀장
치료실 팀장
일반 데스크 직원
일반 치료실 직원
피부관리실 직원
한약사
탕전 담당자
```

## 5.2 사용자 그룹

초기 권한 구조는 단순하게 유지한다.

```text
관리자
팀장
직원
```

추후 필요 시 다음 권한을 검토할 수 있다.

```text
조회 전용
수습 직원
```

단, 권한 구조를 과도하게 세분화하지 않는다.
김포365OS는 운영 편의를 위해 만드는 시스템이지, 복잡한 권한 관리 시스템이 아니다.

## 5.3 핵심 사용자 단위: 마감담당자

김포365OS의 초기 핵심 사용자 단위는 `마감담당자`이다.

마감담당자는 각 파트에서 당일 재고관리 등 마감 관련 필수 업무를 수행하는 담당자를 의미한다.

여기서 `마감담당자`는 사전에 지정된 개인이 아니라 **그날 실제로 해당 업무를 수행한 직원**을 뜻한다.
(Checklist Module에서도 항목을 미리 개인에게 배정하지 않고, 그날 완료한 사람을 completed_by로 기록한다.)

운영 단위는 부서 단위로 관리하되, 실제 기록은 수행자 개인 단위로 남긴다.

```text
운영 단위: 부서(Department)
기록 단위: 개인 계정
책임 확인 단위: 당일 실제 수행자 (사전 지정 아님)
```

재고관리 업무는 데스크, 피부관리실, 치료실 등 전 파트에 이미 매일 배정되어 있으며, 각 파트의 마감담당자가 수행하는 필수 프로세스이다.

따라서 Inventory Module은 김포365OS 초기 단계에서도 전 직원 또는 전 파트가 매일 시스템에 접속할 이유를 제공한다.

## 5.4 계정 및 접속 모델

김포365OS는 개인 계정 사용을 원칙으로 한다.

부서 계정 또는 공용 계정은 사용하지 않는다.

공용 계정을 금지하는 이유는 다음과 같다.

```text
확인 기록의 개인 귀속이 불가능함
체크리스트 완료 기록의 책임 주체가 불명확해짐
공지 확인 여부를 개인 단위로 확인할 수 없음
향후 교육 이수, 내부 요청, 결재 기능 확장에 제약이 생김
```

공용 PC를 사용하더라도 각 직원은 개인 계정으로 로그인한다.

모바일 개인 접속은 허용한다.
다만 접속은 원내 무선 네트워크에 연결된 상태에서만 가능해야 한다.

원외 접속은 차단한다.

초기 접속 모델은 다음과 같다.

```text
계정 원칙: 개인 계정
부서 계정: 금지
공용 계정: 금지
공용 PC 사용: 가능
공용 PC 로그인 방식: 개인 계정 로그인
개인 모바일 접속: 허용
접속 가능 네트워크: 원내 무선 네트워크
원외 접속: 차단
```

## 7. 초기 모듈 구성

## 7.1 Module 1: Inventory Module

현재 구현 완료된 첫 번째 모듈이다.

기존 `gimpo365-inventory` MVP를 기반으로 포함한다.

주요 기능은 다음과 같다.

```text
로그인/권한
입고 등록
출고 등록
초기재고 입력
실사조정 요청/승인
거래이력 조회
재고현황 조회
주문 장바구니
선택 항목 주문
주문 목록
주문 상세
주문서 기반 입고등록
부분입고
미입고 잔여마감
관리품목 상세
공급업체 상세
거래 상세
관리자 리포트
엑셀 다운로드
운영기록 초기화 command
기준정보 점검 command
관리자 > 기준정보 점검 화면
기준정보 엑셀 다운로드
```

Inventory Module은 김포365OS 안에서 다음 위치에 둔다.

```text
운영관리 > 재고관리
```

## 7.2 Module 2: Notice Module

공지사항 모듈이다.

예정 기능은 다음과 같다.

```text
공지 등록
공지 수정
공지 삭제 또는 비활성화
공지 목록 조회
공지 상세 조회
중요 공지 표시
직원 공지 확인 여부 기록
```

공지사항 모듈의 기준 기록은 김포365OS에 남긴다.

카카오톡 등 외부 메신저는 즉시 알림 채널로 병행할 수 있다.
다만 공지의 원문, 최신본, 확인 여부, 운영 기록은 김포365OS가 담당한다.

즉, 외부 메신저는 보조 알림 수단이며, 공지 관리의 기준 시스템은 김포365OS이다.

## 7.3 Module 3: Checklist Module

오픈/마감 체크리스트 모듈이다. **Phase 3(v1) 구현 완료** — 상세는 `docs/modules/checklist/CHECKLIST_*.md`.

Checklist v1은 개인별 할 일이 아니라 **부서 단위 반복업무 수행 기록**이다.

```text
- 부서별 daily 정기업무 (frequency 필드는 daily/weekly/monthly, v1 실제 실행은 daily만)
- 항목별 시기(timing: 오픈/특정 시점/마감, 기본 특정 시점)로 오늘 화면·누락 현황을 정렬
- 오늘 화면은 미완료 항목을 상단, 완료 항목을 하단에 배치 (완료/취소 후 자동 재배치)
- 그날 실제 수행자가 직접 완료/취소 (개인 담당자 사전 지정 없음, completed_by/완료시각 기록)
- 같은 부서 구성원이 완료 상태를 공유 (하루 1레코드)
- TEAM_LEADER 이상 누락 현황 확인 (TEAM_LEADER 본인 부서 / MANAGER·ADMIN 전체 활성 부서)
- 승인/반려 없음 (완료/미완료만)
- 항목 정의·부서 배정은 Django admin으로 초기 설정 (감사 필드는 자동 기록·읽기 전용, 배정 부서 목록 표시)
```

후순위(미구현): weekly/monthly 실행 규칙, 알림·독촉, 완료율 통계, 파일첨부, 별도 관리 UI.

## 7.4 Module 4: SOP / Manual Module

업무 매뉴얼 및 SOP 모듈이다.

예정 기능은 다음과 같다.

```text
업무 매뉴얼 목록
부서별 매뉴얼
상황별 응대 기준
업무 절차 문서
검색
최신본 관리
```

SOP/매뉴얼은 외부 시스템에서 이관하는 자료가 아니라, 김포365OS 내부에서 신규로 작성하고 관리한다.

교육자료 허브와의 연결은 김포365OS 내부 모듈 간 연결로 한정한다.

SOP / Manual Module은 직원이 업무 기준을 확인하고, 신규 직원이 표준 업무 절차를 학습할 수 있는 내부 기준 문서 저장소 역할을 한다.

## 7.5 Module 5: Internal Request / Approval Module

내부 요청 및 결재 모듈이다.

예정 기능은 다음과 같다.

```text
비품 요청
시설/장비 요청
업무 개선 요청
관리자 확인
처리상태 변경
처리 완료 기록
```

초기에는 복잡한 전자결재 시스템을 만들지 않는다.
간단한 요청 등록과 처리상태 확인부터 시작한다.

## 7.6 Module 6: Attendance / Work Schedule Module

근태 및 근무표 모듈이다.

예정 기능은 다음과 같다.

```text
근무표 확인
휴가 일정 확인
근무 변경 요청
관리자 확인
```

근태/급여/노무와 직접 연결되는 기능은 초기 구현 대상에서 제외한다.
초기에는 근무표 확인과 휴가 일정 공유 수준부터 검토한다.

placeholder 화면에서는 명확히 준비 중 상태를 표시한다.

## 11. 공통 운영 원칙

김포365OS의 개발 및 운영 원칙은 다음과 같다.

```text
1. 실제 한의원 운영에 필요한 기능만 만든다.
2. 자동화보다 유지보수 가능성을 우선한다.
3. 직원 교육 난이도가 높은 기능은 후순위로 둔다.
4. 운영 데이터는 삭제하지 않고 기록으로 남긴다.
5. 각 모듈은 작게 만들고, 안정화 후 확장한다.
6. 관리자 편의보다 현장 입력 누락 방지를 우선한다.
7. 기능 추가보다 기준정보 정리와 프로세스 안정화를 우선한다.
8. 차트 프로그램으로 해결 가능한 환자 관련 업무는 가급적 제외한다.
9. 복잡한 기능보다 전 직원이 매일 사용할 수 있는 단순한 구조를 우선한다.
10. 신규 직원이 봐도 이해할 수 있는 화면과 문서를 지향한다.
11. 모든 내부 운영 기능은 OS 내부에 자체 구현한다.
12. 외부 도구로의 링크 허브나 단순 이관 방식에 의존하지 않는다.
13. 각 모듈은 외부 도구 없이 OS만으로 해당 업무를 완결할 수 있도록 설계한다.
14. 제외하지 않은 내부 운영 기능은 OS 내에서 자체 완결한다.
```

김포365OS는 단독으로 한의원 내부 운영을 수행할 수 있어야 한다.

공지, 체크리스트, SOP, 내부 요청 등 김포365OS의 범위에 포함된 내부 운영 기능은 Notion, Jandi, 외부 문서 링크, 외부 협업툴에 의존하지 않고 OS 내부에서 자체 구현한다.

외부 도구는 필요한 경우 보조 알림 채널 또는 임시 운영 수단으로 사용할 수 있다.
그러나 기준 기록, 확인 여부, 최신본, 처리상태는 김포365OS 내부에 남긴다.

이 원칙은 8장의 제외 범위와 구분한다.
환자 진료기록, 예약, 차트, 청구, CRM 등 제외한 영역은 김포365OS가 직접 구현하지 않는다.
반대로 제외하지 않은 내부 운영 기능은 OS 내부에서 자체 완결하는 것을 원칙으로 한다.

## 12. 데이터 운영 원칙

김포365OS의 데이터 운영 원칙은 다음과 같다.

```text
운영 데이터는 삭제보다 기록 보존을 우선한다.
오류는 수정 덮어쓰기보다 취소/반려/철회/상태 변경 이력으로 남긴다.
관리자가 추적할 수 없는 데이터 변경을 만들지 않는다.
위험 command는 기본 dry-run을 원칙으로 한다.
운영 DB와 리허설 DB를 분리한다.
개발/리허설 환경에서 운영 DB를 직접 사용하지 않는다.
운영 DB는 일 1회 이상 자동 백업한다.
백업본은 운영 PC 외 별도 위치 또는 별도 매체에 보관한다.
원내 단일 PC 장애로 운영 이력이 소실되지 않도록 한다.
```

운영/리허설 DB 분리 원칙은 유지한다.

운영 DB 백업은 김포365OS의 필수 운영 정책이다.
운영 DB가 원내 PC 한 대에만 존재하면 PC 장애, 저장장치 장애, 실수 삭제 시 운영 이력이 소실될 수 있다.

따라서 운영 DB는 최소 일 1회 이상 자동 백업하고, 백업본은 운영 PC 외 별도 위치 또는 별도 매체에 보관한다.

Inventory Module의 경우 기존 원칙을 유지한다.

```text
현재고 저장 필드 추가 금지
현재고 직접 수정 금지
StockTransaction 직접 create 금지
거래 상태 직접 변경 금지
입고/출고/초기재고/실사조정은 service 계층 사용
주문 상태 변경만으로 현재고 변경 금지
주문은 재고 증감이 아님
실제 재고 증가는 StockTransaction IN으로만 발생
잔여마감은 재고 증감 아님
운영 데이터 삭제 금지
```
