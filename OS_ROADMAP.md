# OS_ROADMAP.md 수정 지시서 (v0.1 → v0.2)

## 0. 목적

로드맵 검토 결과 확정된 방향 2건과 수용된 피드백 5건을 반영하여 `OS_ROADMAP.md`를 개정한다.
각 항목의 **수정 지시**대로만 반영하고, 마지막 **유지 사항**은 건드리지 않는다.
문서의 원칙 나열식 어투와 Phase 구조는 유지한다. Phase 번호는 뒤에서 밀지 않고 **Phase 1.5만 삽입**한다(Phase 2~7 번호 유지).

## 1. Notice-first를 "패턴 확립용 선택"으로 확정 (7.1, 8.1, 2장·4장 주석)

**확정된 의미**
- Notice-first는 "가장 중요한 모듈이라서 먼저"가 아니다.
- Notice-first는 "가장 단순한 문서형 CRUD로 모듈 추가 패턴(문서 → 앱 → migration → QA)을 먼저 확립하기 위해서"다.
- Checklist가 실질적인 일일 사용 핵심 모듈이다.

**수정 지시**
- 7.1(공지사항 목적)에 다음 취지를 명시한다: "Notice 모듈은 제목·본문·중요/활성 플래그·분류로 구성된 가장 단순한 문서형 CRUD이며, 이후 SOP/매뉴얼이 재사용할 모듈 추가 패턴을 확립하기 위한 최소 모듈이다. 실질적인 일일 사용 핵심 모듈은 Checklist이다."
- 8.1(체크리스트 목적)에 다음 가드 문장을 추가한다: "Notice가 비대해지거나 Checklist를 지연시키지 않도록 한다. Notice의 후순위 기능(확인 독촉, 첨부, 예약 게시 등)은 Checklist 정착 이후에 검토한다."
- 2장 또는 4장에, Phase 2 Notice에서 가장 단순한 문서형 CRUD 패턴을 먼저 확립하고, 이 패턴은 Phase 4 SOP/Manual에서 재사용한다. Phase 3 Checklist는 실질적인 일일 사용 핵심 모듈이므로 Notice v1 이후 지연 없이 착수한다.

## 2. Phase 1.5 신설 — Department/Team 소속 기준 확정 (신규 절 + 4장·13장·14장 갱신)

**성격**: 모델 이전이나 대규모 migration이 아니라, 기존 실체 확인과 공통 기준 결정 단계다.

**수정 지시 — Phase 1과 Phase 2 사이에 "Phase 1.5. Department/Team 소속 기준 확정" 절을 신설한다.**

주요 작업:
```text
기존 Inventory MVP에 Department/유사 소속 모델이 실재하는지 코드로 확인한다.
있으면 OS 공통 소속 기준으로 승격/공유할지 결정한다.
없으면 core에 Department/Team 기준정보를 신설할지 결정한다.
필요 시 User와 Department/Team 연결(FK)을 정리한다.
물리적 앱 이동이나 테이블 rename은 서두르지 않는다.
```

완료 기준:
```text
기존 Department 모델 존재 여부가 확인되어 문서에 기록됨
OS 공통 소속 기준(core 관리)을 사용할지 결정됨
User–Department/Team 연결 방식이 결정됨(추가는 additive 범위로 한정)
파괴적 migration 없이 처리됨
python manage.py check / test 통과
```

원칙:
```text
이 단계의 변경은 User FK 추가 + Department/Team 테이블 추가 등 additive 범위로 한정한다.
컬럼 rename, 앱 물리 이동 등 파괴적 변경은 이 단계에서 하지 않는다.
소속 모델은 특정 모듈이 아니라 OS 공통 기준정보로 설계한다(Checklist 전용으로 좁게 만들지 않는다).
```

**연동 갱신**
- 4장 전체 개발 단계 요약에 `Phase 1.5. Department/Team 소속 기준 확정`을 Phase 1과 Phase 2 사이에 삽입한다.
- 13장 의존성 테이블: Phase 3(체크리스트)의 선행 조건을 "Department/Team 소속 모델 원칙" → "**Phase 1.5 완료(Department/Team 소속 기준 확정)**"로 변경한다. Phase 1.5 행(선행: Phase 1 OS 틀 + 기존 Department 확인)을 추가한다.
- 14장 migration 리스크 테이블에 Phase 1.5 행 추가: Migration 가능성 **낮음**, 리스크 "User FK + Department/Team 테이블 추가(additive, reversible)".

## 3. 모듈 정착 루프 도입 (신규 절 + 각 모듈 Phase 완료 기준 갱신)

**문제**: 현재 완료 기준은 전부 기술 기준(check/test/smoke)이고, 실사용·채택은 Phase 7로 통째로 미뤄져 있다. 로드맵이 선형이라 이전 모듈로 되돌아가 조정하는 지점이 없다.

**수정 지시**
- "모듈 정착 루프"를 정의하는 신규 절을 추가한다(15장 리허설 원칙 근처 권장):
```text
각 업무 모듈(Phase 2~6)은 기술 완료 후 다음 정착 루프를 거친 뒤 다음 Phase로 넘어간다.
1. 운영 반영
2. 직원 안내 및 교육
3. 1~2주 실사용 관찰
4. 관찰 결과에 따른 조정
이전 모듈로 돌아가 조정하는 것은 예외가 아니라 정상 흐름이다.
```
- Phase 2~6 각 절의 완료 기준 끝에 한 줄을 공통 추가한다: "기술 완료 후 모듈 정착 루프(운영 반영 → 직원 안내/교육 → 1~2주 실사용 관찰 → 조정)를 거친다."
- 2장 로드맵 기본 원칙에 항목 추가: "모듈은 기술 완료가 아니라 실사용 정착까지를 완료로 본다."

## 4. Phase 1 문서 부담 분리 (6.2, 6.5)

**문제**: Phase 1이 OS 셸 구축과 문서 6종 신규 작성을 한 단계에 묶어 과적재 상태다. 셸이 눈에 보이기 전 문서 완성에 시간이 묶일 위험.

**수정 지시** — 6.2 주요 작업에서 문서 작성을 두 부류로 구분한다.
- **셸 코드 이전 필수 문서(Phase 1 완료 조건)**: `OS_WORKING_RULES.md`, `OS_DB_OPERATIONS.md`, `OS_OPERATIONS_SETUP.md`.
- **stub 후 빌드하며 성장(완료 조건 아님)**: `OS_TECH_SPEC.md`, `OS_TASKS.md`, `OS_MANUAL_QA_CHECKLIST.md`. 초기에는 stub만 두고 셸 구현과 함께 채운다.
- 6.5 완료 기준에서 "문서 6종 완성"을 제거하고, "필수 문서 3종 작성 + 나머지 3종 stub 생성"으로 대체한다.
- `OS_WORKING_RULES.md`는 문서 우선순위 1위이자 Claude Code 최상위 구속 문서이므로 Phase 1의 우선 산출물로 명시한다.

## 5. OS MVP 컷라인 명시 (신규 절, 4장/20장과 연동)

**수정 지시** — "OS MVP 컷라인"을 정의하는 신규 절을 추가한다(4장 직후 권장):
```text
OS MVP 컷라인:
Phase 1 OS 최소 틀 + Phase 1.5 Department/Team 소속 기준 + 재고관리(Module 1) + 공지사항(Phase 2) + 체크리스트(Phase 3)

이 컷라인까지 도달하면 김포365OS는 핵심 목적(전 직원 일일 업무 허브)을 달성한 것으로 본다.
SOP/매뉴얼(Phase 4), 내부 요청(Phase 5), 근태/근무표(Phase 6)는 이 위에 얹는 가산 모듈이다.
개발이 Phase 5 중간에 중단되더라도, MVP 컷라인 도달 상태는 '핵심 완성 + 추가 진행 중'이라는 안전한 착지 지점이다.
```
- 20장 성공 기준에 "MVP 컷라인(재고+체크리스트+공지) 도달"을 초기 성공의 기준선으로 한 줄 추가한다.

## 6. Phase 4 SOP — 그릇과 콘텐츠 분리 (9.1, 9.5)

**문제**: 매뉴얼 모듈(그릇)은 문서형 CRUD라 쉽지만, SOP를 실제로 저작하는 작업은 별개의 상시 프로젝트다. 구분이 없으면 콘텐츠 저작에 발목 잡혀 Phase 4가 끝나지 않는다.

**수정 지시**
- 9.1에 명시한다: "이 Phase의 범위는 SOP/매뉴얼을 담고 관리하는 모듈(그릇)의 구현이다. 실제 SOP 콘텐츠 저작은 상시 진행 작업이며 Phase 완료를 막지 않는다."
- 9.5 완료 기준을 "모듈이 작동하고, 최소 샘플 매뉴얼로 작성·조회·최신본 관리가 검증됨" 수준으로 한정한다(전체 SOP 저작 완료를 완료 조건으로 삼지 않는다).

## 7. Phase 5 내부 요청 — 상태값 축소 (10.4)

**문제**: 상태값 6개(요청됨/확인중/처리중/처리완료/보류/반려)는 "간단히 시작"이라는 10.1 취지에 과하다. 전이 로직·UI가 그만큼 늘어난다.

**수정 지시**
- 10.4 초기 상태값을 다음으로 축소한다: `요청됨 → 처리중 → 처리완료`, 그리고 `반려`(3+1).
- `확인중`, `보류`는 실제 운영에서 필요성이 확인되면 추가하는 후순위 상태로 명시한다.
