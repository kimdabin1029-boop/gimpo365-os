# gimpo365-inventory v0.1 수동 점검 체크리스트 (MANUAL QA)

이 문서는 **자동 테스트로 검증하지 않는 항목**을 운영 투입 전에 사람이 직접 점검하기 위한
체크리스트다. 자동 테스트(P0, PostgreSQL 147건)는 별도로 통과되어 있어야 하며, 이 문서는
화면 카피·안내 문구·UX·운영 데이터·모바일 사용성·타이밍 기반 동시성 등 자동화에 적합하지
않은 부분을 다룬다. (TASKS.md TASK 20 / TECH_SPEC §15 수동·통합 검증)

## 사용 방법

- 각 항목을 실제 화면에서 직접 수행하고 `[ ]` → `[x]` 로 표시한다.
- "예상 결과"와 다르면 **비고**에 현상을 적고 운영 투입을 보류한다.
- 점검 환경(서버 URL/브라우저/일시/점검자)을 아래에 기록한다.

```text
점검 일시 :
점검 환경 : (예: https://inventory.example.com / Chrome 126 / iPhone Safari)
점검자    :
빌드/커밋 : (git log --oneline -1)
```

> 표기: 역할 STAFF < TEAM_LEADER < MANAGER < ADMIN. 기본 적용 부서는 피부실(스킨앤라인)/치료실.

---

## 1. 계정 / 권한 점검

- [ ] **활성 ADMIN 계정이 2개 이상** 존재한다. (비상 복구 대비 — PRODUCT_SPEC §14.1)
- [ ] **MANAGER 이상 승인 가능 계정이 2개 이상** 존재한다. (승인 병목 방지 — PRODUCT_SPEC §11.6)
- [ ] STAFF 로 로그인 → 대시보드로 진입하고 본인 부서 정보가 보인다.
- [ ] TEAM_LEADER 로 로그인 → 본인 부서 범위로 동작한다.
- [ ] MANAGER 로 로그인 → 전체 부서 데이터에 접근할 수 있고, 승인 대기 큐 링크가 보인다.
- [ ] ADMIN 으로 로그인 → 일반 운영 화면 + 네비게이션에 **Django Admin 버튼**이 보인다.
- [ ] STAFF / TEAM_LEADER / MANAGER 에게는 **Django Admin 버튼이 보이지 않는다.** (is_staff=False)
- [ ] 비활성(is_active=False) 사용자로는 로그인할 수 없다.
- [ ] 로그아웃이 정상 동작하고, 로그아웃 후 보호 화면 접근 시 로그인으로 이동한다.
- [ ] 비로그인 상태에서 `/inventory/...` 접근 시 로그인 화면으로 redirect 된다.

## 2. 기본 데이터 점검

- [ ] **피부실(스킨앤라인)** 부서가 생성되어 있고 `active_for_inventory=True` 다.
- [ ] **치료실** 부서가 생성되어 있고 `active_for_inventory=True` 다.
- [ ] 탕전실은 재고관리 대상이 아니다(`active_for_inventory=False`) — 관리품목 선택 목록에 안 보인다.
- [ ] 기본 **품목 마스터(Item)** 가 Admin 에서 등록 가능하다. (분류 고정 선택값 동작)
- [ ] 기본 **공급업체(Supplier)** 가 Admin 에서 등록 가능하다.
- [ ] **부서별 관리품목(ManagedItem)** 등록 흐름이 동작한다. (Item + Department + 단위 + 최소재고)
- [ ] 같은 부서에 같은 품목을 중복 등록하려 하면 막힌다.
- [ ] 단위(unit)·분류(category)는 드롭다운 고정 선택값으로만 입력된다(자유입력 불가).
- [ ] 보관장소(storage_location)는 자유입력이며 현재고 화면에서 부분일치 검색이 동작한다.

## 3. 입고 / 출고 흐름 점검

- [ ] STAFF 가 본인 부서 관리품목으로 **입고 등록**을 할 수 있다. (즉시 반영)
- [ ] STAFF 입고 화면에는 **입고단가(unit_price) 필드가 보이지 않는다.**
- [ ] TEAM_LEADER 이상 입고 화면에는 입고단가 필드가 보인다.
- [ ] 입고 시 공급업체 기본값이 관리품목의 기본 공급업체로 채워지고, 변경할 수 있다.
- [ ] STAFF 가 **출고 등록**(사용/폐기/분실/증정/기타출고)을 유형별로 구분해 등록할 수 있다.
- [ ] **현재고를 초과하는 출고**를 시도하면 저장되지 않고 오류 메시지가 뜬다.
- [ ] 현재고와 **같은 수량 출고**는 허용된다(현재고 0 가능).
- [ ] STAFF / TEAM_LEADER 는 **타 부서 관리품목**을 선택할 수 없다.
- [ ] 입고/출고 저장 후 **같은 화면에 머무르며**(연속 입력) 성공 메시지가 표시된다.
- [ ] 현재고 조회 화면 수치가 입고/출고 결과와 일치한다.

## 4. 최초 재고 입력 / 실사조정 흐름 점검 (v0.1.1 통합)

- [ ] **실사조정 요청은 TEAM_LEADER 이상**만 가능하다(STAFF 메뉴 미노출, URL 접근 403).
- [ ] 별도 '초기재고 입력' 메뉴가 없고, **'실사조정 요청' 화면에서 통합 처리**된다.
- [ ] 승인된 최초 재고가 **없는** 품목 선택 → 화면이 **'최초 재고 입력'** 모드로 안내(사유 불필요).
- [ ] 승인된 최초 재고가 **있는** 품목 선택 → **'실사조정'** 모드로 안내(사유 필수).
- [ ] TEAM_LEADER 요청은 PENDING, MANAGER/ADMIN 요청은 즉시 처리(초기재고는 APPROVED).
- [ ] 이미 APPROVED 최초 재고가 있으면 추가 최초 재고를 만들 수 없다(자동으로 실사조정으로 처리).
- [ ] 요청은 승인 전 현재고에 반영되지 않는다.
- [ ] 실사조정 내역에 최초 재고 입력/실사조정이 거래유형으로 구분 표시된다.
  (실사조정: 시스템 수량 / 실제 수량 / 차이, 최초 재고 입력: 기준 없음 / 입력 수량 / 반영 수량)

## 5. 승인 / 반려 / 철회 / 취소 흐름 점검

- [ ] MANAGER 가 승인 대기 큐에서 PENDING 거래를 **승인**하면 현재고에 반영된다.
- [ ] MANAGER 가 **반려**하면 현재고에 반영되지 않고 반려 사유(review_note)가 남는다.
- [ ] 생성자가 본인 PENDING 거래를 **철회**할 수 있고, 철회 사유가 남는다.
- [ ] 타인의 PENDING 거래는 철회할 수 없다.
- [ ] **초기재고 일괄 승인**: 여러 PENDING 초기재고를 선택해 한 번에 승인할 수 있다.
- [ ] 일괄 승인 결과에 "승인 완료 N건 / 승인 제외 M건"이 메시지로 표시된다(부분 성공).
- [ ] STAFF 가 **본인이 당일 등록한 APPROVED 일반 거래**를 취소할 수 있다(취소 사유 필수).
- [ ] STAFF 가 타인 거래/전일 거래는 취소할 수 없다(취소 버튼 미표시 + 직접 URL 접근 차단).
- [ ] TEAM_LEADER 는 본인 부서 당일 일반 거래를 취소할 수 있다.
- [ ] MANAGER/ADMIN 은 전체 APPROVED 일반 거래를 취소할 수 있다.
- [ ] **INITIAL_COUNT / ADJUSTMENT 에는 취소 버튼이 없고** 직접 URL 접근도 차단된다.
- [ ] 취소 결과 현재고가 음수가 되는 경우 취소가 차단된다.
- [ ] 거래 이력에 입력자/승인자/취소자/사유가 남는다(삭제가 아니라 상태 이력으로 보존).

## 6. Admin 제한 점검

- [ ] STAFF / TEAM_LEADER / MANAGER 는 `/admin/` 에 접근할 수 없다(로그인 화면으로 redirect).
- [ ] ADMIN 만 `/admin/` 에 접근할 수 있다.
- [ ] Django Admin 에서 **거래 원장(StockTransaction) 추가(add)가 불가능**하다.
- [ ] Django Admin 에서 **거래 원장 삭제(delete)가 불가능**하다.
- [ ] 거래 원장의 managed_item / status / 수량 / 감사 필드가 **readonly** 로 보인다(수정 불가).
- [ ] 운영 개시(APPROVED 거래 존재) 후 ManagedItem 의 **단위(unit) 변경이 차단**된다.
- [ ] 마스터 데이터(사용자/부서/품목/관리품목/공급업체)는 삭제보다 비활성화로 운영한다.

## 7. 화면 문구 / UX 점검

- [ ] 모든 주요 화면 제목/라벨이 한국어로 자연스럽게 표시된다(깨짐 없음).
- [ ] 입고/출고/실사조정/초기재고 **성공 메시지**가 명확하다.
- [ ] 권한/현재고/유일성 위반 시 **실패 메시지**가 사용자에게 이해되게 표시된다.
- [ ] 초기재고/실사조정 보정 안내 문구(승인 전 미반영 등)가 표시된다.
- [ ] 연속 입력 UX: 입고/출고 저장 후 폼이 비워진 채 같은 화면에 남아 바로 다음 입력이 가능하다.
- [ ] 거래 이력 정렬이 발생일시→입력일시→id 역순으로 보인다.
- [ ] 최소재고 이하 품목 화면에서 부족 수량이 올바르게 계산되어 보인다.

## 8. 모바일 사용성 점검

- [ ] 모바일 브라우저(예: iPhone Safari / Android Chrome)에서 로그인할 수 있다.
- [ ] 모바일에서 입고/출고 등록 폼을 사용할 수 있다(필드 입력/제출 가능).
- [ ] 모바일에서 현재고/거래 이력 표가 가로 스크롤 등으로 확인 가능하다.
- [ ] MANAGER 가 모바일/외부 환경에서 승인 대기 큐를 열고 승인/반려할 수 있다. (PRODUCT_SPEC §11.6)

## 9. 동시성 수동 / 통합 점검

> 타이밍 기반 경합은 자동 테스트로 안정적으로 재현하기 어려워 수동/통합으로 분리한다.
> 결정적 보장(중복 INITIAL_COUNT 방지)은 자동 테스트(4.7 DB 유니크 / 10.6 승인 시점 재검사)로 커버된다.

- [ ] **동시 출고**: 두 사용자가 같은 품목을 거의 동시에 출고해도 현재고가 음수가 되지 않는다.
- [ ] **approve / reject 동시**: 같은 PENDING 거래에 승인·반려가 동시에 들어와도 한쪽만 적용되고 다른 쪽은 차단된다.
- [ ] **approve / withdraw 동시**: 같은 PENDING 거래에 승인·철회가 동시에 들어와도 이중 처리되지 않는다.
- [ ] **동시 INITIAL_COUNT 승인**: 같은 관리품목의 두 PENDING 초기재고를 동시에 승인 시도해도 APPROVED 는 1건만 남는다.
- [ ] 위 경합 상황에서 오류 메시지가 사용자에게 적절히 표시된다(중복/충돌 안내).

## 10. 운영 투입 최종 승인

- [ ] 1~9 섹션의 필수 항목이 모두 통과했다.
- [ ] 자동 P0 테스트가 **PostgreSQL** 에서 전부 통과함을 확인했다(SQLite 미사용).
- [ ] 활성 ADMIN 2개 이상 / 승인 가능 MANAGER 이상 2개 이상이 준비됐다.
- [ ] 적용 파트(피부실 또는 치료실)의 관리품목·초기재고가 등록·승인 완료됐다.
- [ ] 팀장 외 1명 이상이 입고/출고/조회/오입력 취소 절차를 익혔다. (실사조정/최초 재고 입력은 팀장 이상)
- [ ] 비상 ADMIN 복구 절차가 운영진 문서에 기록되어 있다(일반 직원 비공개).

```text
최종 승인자 :
승인 일시   :
비고        :
```
