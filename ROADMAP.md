# gimpo365-inventory 로드맵
> 문서 범위: 김포365OS Module 1 — Inventory

문서 상태: 방향성 정리. 구조 분리 원칙은 [ARCHITECTURE.md](ARCHITECTURE.md) 참고.

> 핵심: **지금은 재고관리 안정화가 최우선.** 아래 확장 모듈은 후보일 뿐, 현재 구현하지 않는다.

## 현재 우선순위

```text
1. 재고관리 핵심 기능 안정화
2. 원내 알파테스트
3. 직원 피드백 반영
4. v0.1.1 사용성 개선 (재고관리 입력 UX / 안정화)
5. 그 이후 확장 검토
```

## 단계별

### v0.1 (완료)
- 재고관리 핵심: 입고/출고/폐기/분실/증정/기타출고, 초기재고, 실사조정, 승인/반려/철회/취소,
  현재고/최소재고 이하/거래 이력, Django Admin. 자동 테스트 PostgreSQL 통과.

### v0.1.1 (진행)
- 재고관리 **사용성 개선 + 안정화** (입력 UX, 화면 정리, 계정 셀프서비스 등).
- 원내 알파테스트 및 피드백 반영.
- 확장 기능은 넣지 않는다.

### v0.2 이후 (후보, 미확정)
- 재고 도메인 보강 후보(발주/유통기한 알림/정기 실사 등)는 PRODUCT_SPEC §17 참고.
- 인증 보강 후보(로그인 실패 잠금 등)는 TASKS.md v0.2 메모 참고.

#### 운영요청(Request) 기능 (후보, 미구현)
직원이 관리자에게 요청사항을 전달하는 기능. **이번(v0.1.3) 작업에서는 구현하지 않는다.**
(모델/마이그레이션/화면/URL 모두 만들지 않음 — 아이디어 보존용 기록.)

```text
예상 용도:
- 품목 사용중지 요청 / 신규 품목 등록 요청 / 공급업체 등록 요청
- 시스템 오류 제보 / 기능 개선 요청

예상 필드:
- 요청자 / 부서 / 요청 유형 / 요청 내용
- 상태: 접수 / 처리중 / 완료 / 반려
- 관리자 답변 / 처리자 / 처리일시
```

- 이 기능은 Inventory 전용이 아니라, 향후 **gimpo365OS 공통 "운영요청" 모듈** 후보로 둔다.
- 현재는 inventory 안에 별도 모델/앱을 만들지 않는다. (ARCHITECTURE 모듈 분리 원칙 유지)

### 장기 (인트라넷 / 원내 업무 포털 — 후보, 미착수)
화면상 하나의 포털처럼 보이되 **모듈별 별도 앱**으로 분리한다. (ARCHITECTURE §2~4)

```text
board       원내 공지사항 / 게시판
docs        업무 매뉴얼 / SOP / 교육자료
checklists  일일점검 / 마감점검 / 장비점검
leave       연차신청 / 휴가승인
attendance  근태기록
hr          면담기록 / 인사메모 / 평가자료
```

`accounts`(계정/역할/재직상태)와 `core`(부서/공통)는 이들 모듈의 공통 기반으로 재사용한다.

## 지금 하지 않는 것 (명시)

```text
- 게시판 앱 구현 안 함
- 인사관리 앱 구현 안 함

- inventory 모델을 범용 게시판/업무 구조로 변경 안 함
- accounts.User 대규모 변경 안 함
- core.Department 를 깨는 migration 안 함
```
